# Zenoh

Zenoh exposes two communication patterns from a session: publish/subscribe and query/reply. In pub/sub, a publisher writes samples under a key and subscribers receive samples for matching key expressions. In query/reply, a querier requests data for a key expression and queryables answer with replies. Keys are slash-separated resource paths such as `robot/sensor/temp`; key expressions add matching with wildcards such as `robot/sensor/*` or `robot/**`.

The session owns the runtime connection to the Zenoh network and declares publishers, subscribers, queriers, and queryables. A node can run as a client that connects to another Zenoh node, a peer that participates directly in the graph, or a router that forwards traffic between clients, peers, and network segments. Forwarding is driven by declared interest in key expressions, so traffic is routed toward matching subscribers or queryables rather than broadcast blindly. The Rust crate also exposes handlers for receiving samples or replies through channels or callbacks, plus optional features for transports, authentication, shared memory, compression, liveliness, matching status, and DDS interoperability.
